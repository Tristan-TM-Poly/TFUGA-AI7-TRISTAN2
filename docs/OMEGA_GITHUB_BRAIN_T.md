# Ω-GITHUB-BRAIN-T — GitHub as a Living Knowledge Brain

**Status:** OAK-safe canonical scaffold.  
**Scope:** repository knowledge architecture, living canon, theory cards, repository genome, green tensor, and repair atlas.  
**Non-goals:** no external execution, no secret access, no autonomous publication, no claim that metaphors are biological facts.

## 0. Core metaphor and boundary

Ω-GITHUB-BRAIN-T treats GitHub as an executable memory organ for the Tristan ecosystem.

```text
Repository = lobe
Pull request = neuron
Commit = synapse
Test = reflex
Workflow = circuit
Issue = stimulus
Benchmark = measurement reflex
OAK report = judgment layer
M+ = useful success memory
M- = useful negative memory
```

OAK boundary: this is an architecture metaphor and an operational scaffold. It does not claim that a GitHub repository is literally a biological brain.

## 12. Green-by-construction rules

A repository becomes healthier when each addition includes:

```text
canon_delta
schema_or_manifest_delta
test_or_validator_delta
oak_decision
m_plus_or_m_minus_update
rollback_note
```

A change is not green if it hides checks, weakens validation, removes uncertainty, or bypasses provenance.

## 13. M+ seeds

- PR #210 pattern: a cognitive OS canon can be merged safely when bounded to docs, data, and dependency-free tests.
- PR #209 pattern: OAKBench scoring becomes useful when it blocks sensitive public automation and includes tests.
- PR #25 pattern: validators should distinguish canonical richness from effective bounded validation views.

## 14. M- seeds

- Draft status blocks merge and readiness transitions.
- mergeable=false blocks merge.
- Sensitive scopes block auto-merge even if other checks look green.
- Duplicate blocker reports are noise without new evidence or repair.
- A metaphor must remain marked as metaphor until formalized.
