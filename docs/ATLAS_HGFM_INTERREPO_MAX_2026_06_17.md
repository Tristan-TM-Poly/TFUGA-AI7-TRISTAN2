## 2. Multi-level language ladder

The same artifact should be readable at several levels.

### L0 — Machine bytes / files

Raw repository objects: `.py`, `.json`, `.yml`, `.toml`, `.ipynb`, `.tex`, `.md`, `.jsonl`, workflows, manifests, schemas, reports.

Goal: every file must be indexed by path, type, owner repo, status, dependencies, and nearest DCT packet.

### L1 — Machine semantics / agent-readable graph

Objects become machine-level semantic nodes:

- schemas;
- manifests;
- relation graphs;
- registries;
- benchmarks;
- tests;
- checksums;
- OAK reports;
- M_MINUS residues;
- status labels.

Output target: `ATLAS_HGFM_MACHINE.yml` per repo.

### L2 — Developer/programmer layer

Objects become modules, commands, APIs, test suites, CI jobs and executable examples.

Output target: `ATLAS_CODE.md` per repo.

### L3 — Research layer

Objects become definitions, hypotheses, models, reductions, equations, proofs, simulations, benchmarks, datasets and validation protocols.

Output target: `ATLAS_RESEARCH.md` per repo.

### L4 — Human explanatory layer

Objects become readable maps, summaries, roadmaps, guided tours and examples.

Output target: `ATLAS_HUMAN.md` per repo.

### L5 — Public/pedagogical layer

Objects become demos, educational quest mode, web navigation, publication spine, launch notes, product sheets, public gateway pages.

Output target: `ATLAS_PUBLIC.md` per repo.

### L6 — Canon/OAK layer

Objects become status-tagged claims and promotion gates:

```text
S: speculative
E: exploratory
X: crystallizable
D: locally demonstrated
C: canonical
A: archived
```

Output target: `ATLAS_CANON.md` and `ATLAS_M_MINUS.jsonl` per repo.

### L7 — HGFM inter-repo layer

Objects become a coupled hypergraph:

```text
repo → artifact → DCT → code/proof/test → OAK → M_MINUS → canon → atlas → next generator
```

Output target: `ATLAS_HGFM_INTERREPO.yml`.
