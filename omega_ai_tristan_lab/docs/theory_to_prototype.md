# Theory → Prototype Doctrine

## Format canonique

```text
Nom :

Risques :

```

## Pipeline

```text
idée brute
→ TheoryCard
→ prototype minimal
→ test minimal
→ benchmark
→ OAK report
→ IP classification
→ revenue map
→ next action
```

## Anti-erreurs M⁻

- Ne pas confondre fertilité et preuve.
- Ne pas publier une invention potentiellement brevetable avant classification.
- Ne pas promettre de revenus.
- Ne pas dire “autonome” si une étape externe demande autorisation humaine.
- Ne pas intégrer de code OSS sans licence/provenance.
- Ne pas accepter un résultat IA sans test de contradiction.

## Définition du prototype minimal

Un prototype minimal est le plus petit système qui :

1. reçoit une entrée réelle,
2. produit une sortie observable,
3. peut échouer,
4. possède au moins un test,
5. expose ses limites,
6. propose la prochaine amélioration.

## Critère de canonisation

Une idée devient CANON seulement si elle possède :

- définition stable,
- usage clair,
- test ou preuve partielle,
- limites écrites,
- statut IP/revenu,
- place dans l'écosystème TFUGA/HGFM/CVCD/OAK.
